You are a senior full-stack engineer, AI product builder, and cybersecurity safety reviewer.

Build a complete local proof-of-concept web application that demonstrates how a consent-based AI voice-cloning product could work. This is for a cybersecurity education demo showing how voice-cloning products can be built, what risks they create, and what safeguards should exist before this kind of technology is used in the real world.

The app must allow a host to create a demo session, add participants, collect explicit consent from each participant, record a short live voice sample from each consenting participant, and allow the host to type a sentence that is played back using that participant’s own recorded voice.

The first version should be simple, local, and runnable. Prioritize the complete product flow, safety controls, and clear code comments over advanced UI design.

Non-negotiable safety requirements

1. The app must be consent-based by design.
2. Do not create hidden recording features.
3. Do not record anyone without a visible consent step.
4. Do not allow voice generation unless the participant has explicitly consented.
5. Enforce consent checks in both the frontend and backend.
6. Show clear warnings that generated audio is a synthetic voice clone based on the participant’s recorded voice.
7. Add a delete button and a revoke-consent button so a participant can remove their voice sample and withdraw consent.
8. Store consent status, consent timestamp, and revocation timestamp.
9. Add a safety page explaining that voice cloning can be misused for scams, impersonation, social engineering, fraud, harassment, misinformation, and unauthorized recordings.
10. Add code comments explaining where abuse-prevention controls are enforced.
11. Do not build features for deception, impersonation, bypassing consent, covert recording, or recording people who are not part of the session.
12. Do not include public figure impersonation features.
13. Do not include file uploads for voice data. The app must only accept audio captured live through the browser microphone during an active consent session. There must be no upload endpoint, no drag-and-drop upload, no “paste a link,” and no way to import a pre-existing audio file as a voice sample.
14. Bind the consent phrase and the voice sample to one continuous capture. The participant must record the consent phrase and the additional voice sample back-to-back in one uninterrupted browser recording session. Both clips must be tagged with the same capture_session_id.
15. The backend must reject an audio-sample submission if its capture_session_id does not match a completed consent-phrase recording for that participant.
16. Filter what the host is allowed to generate. Before any generation request reaches the voice provider, run the input text through a content check that blocks or flags categories such as:
    * Financial authorization
    * Payment instructions
    * Wire-transfer requests
    * One-time codes, PINs, or passwords
    * Urgent “send money now” language
    * Attempts to impersonate a real-world public figure
    * Attempts to impersonate someone who is not in the session participant list
    * Requests that sound like fraud, threats, harassment, or social engineering
17. Blocked generation attempts must be logged. Do not silently drop them. The host dashboard must show a running count of blocked generation attempts.
18. Add an embedded spoken disclosure to every generated audio clip, such as: “This is AI-generated audio created for a consent-based cybersecurity education demo.”
19. Auto-expire all session data. Consent records, audio samples, and generated clips must have a configurable TTL. The default TTL should be short, such as 24 hours.
20. When the host ends a session, immediately purge all consent records, audio samples, and generated clips for that session.
21. No voice data should persist indefinitely by default.

Tech stack

Use the following stack:

* Frontend: Next.js with TypeScript
* Backend: FastAPI with Python
* Database: SQLite
* Audio capture: Browser MediaRecorder API only
* Voice generation: Local voice-cloning provider interface with a clearly labeled fallback option

There must be no file input element anywhere in the participant audio-capture flow.

Voice generation requirements

The app should be structured around a provider interface called VoiceCloneProvider.

Create two provider classes:

1. LocalCloneProvider
    * This is the default provider.
    * It should use a local open-source voice-cloning or zero-shot TTS model that can condition generation on a short reference clip.
    * Suggested options include Coqui XTTS-v2, F5-TTS, or OpenVoice v2.
    * The generate() method must take:
        * target text
        * participant reference audio path
        * participant ID
        * session ID
    * Generation must fail if the participant does not have a matching verified reference sample.
    * The provider call must pass the participant’s own stored live-captured audio sample as the speaker/reference clip.
2. BrowserTTSProvider
    * This is only an offline fallback for environments where the local model cannot load.
    * It must be clearly labeled: “Fallback only. This does not clone voices.”
    * It must never be the default provider.
    * The app must show an on-screen warning when the fallback provider is active.

Keep the provider interface flexible so that consent checks, content filtering, logging, and embedded audio disclosure wrap every provider in the same way.

The generated audio must always include the spoken disclosure, regardless of which provider produced it.

Required user flow

1. Host creates a session.
2. Host optionally sets a data-retention TTL. Default is 24 hours.
3. Host manually adds participant names.
4. Each participant receives or opens a unique participant page.
5. Participant reviews a consent disclosure.
6. Participant clicks an explicit consent button before recording.
7. Participant records one continuous browser-microphone session:
    * First, the participant says: “I consent to this voice sample being used for this cybersecurity education demo.”
    * Then, without stopping or uploading anything, the participant records a short additional voice sample.
8. Backend stores the capture_session_id and validates that the consent phrase and sample belong to the same uninterrupted capture flow.
9. Host dashboard shows:
    * Participants
    * Consent status
    * Audio sample status
    * Revocation status
    * Expiration time
    * Blocked generation attempt count
10. Host types a sentence.
11. Host selects a consenting participant.
12. App runs the input sentence through the content filter.
13. If the input is blocked, the host sees the reason and the attempt is logged.
14. If the input passes, the app generates audio using the selected participant’s own reference sample.
15. The generated clip includes the embedded spoken disclosure.
16. App blocks generation if:

* Participant has not consented
* Participant revoked consent
* Participant has no audio sample
* Participant’s audio sample is deleted
* Participant’s capture_session_id is invalid
* Session expired
* Session ended

17. Host can end the session at any time.
18. Ending the session immediately purges all session data, participant consent records, audio samples, and generated clips.

Required backend endpoints

Create these endpoints:

* POST /sessions
* GET /sessions/{session_id}
* DELETE /sessions/{session_id}
* POST /sessions/{session_id}/participants
* POST /participants/{participant_id}/consent
* POST /participants/{participant_id}/revoke-consent
* POST /participants/{participant_id}/audio-sample
* DELETE /participants/{participant_id}/audio-sample
* POST /generate-voice
* GET /health

Important endpoint rules:

* POST /participants/{participant_id}/audio-sample must only accept live browser-recorded audio tagged with a valid capture_session_id.
* Do not create any generic file upload endpoint.
* Do not create any endpoint that imports voice data from a URL, local file path, cloud storage link, or uploaded file.
* POST /generate-voice must run the content filter before calling the voice provider.
* POST /generate-voice must log blocked attempts.
* POST /generate-voice must pass the participant’s own reference sample to the cloning provider.
* DELETE /sessions/{session_id} must immediately purge all data for that session.

Required backend logic

Implement:

1. Consent validation
    * Participant must have active consent.
    * Participant must not have revoked consent.
    * Session must not be expired or ended.
2. Audio sample validation
    * Participant must have a live-recorded audio sample.
    * Audio sample must not be deleted.
    * Audio sample must not be expired.
    * capture_session_id must match a completed capture session for that participant.
3. Content filtering
    * Implement a simple keyword and heuristic-based filter.
    * It is acceptable for the MVP to be a stub, but it must be clearly documented as not production-grade.
    * It must block obvious fraud and impersonation text before the provider is called.
4. Blocked attempt logging
    * Log blocked input text, blocked reason, participant ID, session ID, and timestamp.
5. Data expiration
    * Include cleanup logic for expired sessions, audio samples, and generated clips.
    * Enforce expiration checks before generation.
    * Purge all data when the host ends a session.
6. Provider wrapping
    * The content filter must run before provider.generate().
    * The provider must not be called if the content filter blocks the request.
    * The provider must not be called if consent checks fail.
    * The provider must not be called if reference audio is missing or invalid.
    * Every generated audio clip must include the spoken disclosure.
