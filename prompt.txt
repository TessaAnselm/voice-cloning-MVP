You are a senior full-stack engineer, AI product builder, and cybersecurity safety reviewer.

Build a complete proof-of-concept web application that demonstrates how a consent-based AI voice-cloning product could work. This is for a cybersecurity education video showing how easy it is to create this type of product from a single prompt, and what the security, privacy, and social engineering risks are.

The app should allow a host to create a demo session, add participants, collect explicit consent from each participant, record a short voice sample from each consenting participant, and then allow the host to type a sentence that can be played back as a synthetic voice for that participant.

## Important safety rules (non-negotiable product requirements, not suggestions)

1. This app must be consent-based by design.
2. Do not create hidden recording features.
3. Do not record anyone without a visible consent step.
4. Do not allow voice generation unless the participant has explicitly consented.
5. Enforce consent checks in both the frontend and backend.
6. Add clear warnings that generated audio is synthetic.
7. Add a delete and revoke-consent button so a participant can remove their voice sample and withdraw consent.
8. Store consent status, consent timestamp, and revocation timestamp.
9. Add a safety note explaining that this technology can be misused for scams, impersonation, social engineering, fraud, and misinformation.
10. Add comments in the code explaining where abuse-prevention controls are enforced.
11. Do not build features for deception, impersonation, bypassing consent, or covert recording.
12. **No file uploads for voice data, ever.** The app must only accept audio captured live through the browser microphone, in-app, during an active consent session. There must be no upload endpoint, no drag-and-drop, no "paste a link," and no way to import a pre-existing audio file as a voice sample. This is the control that prevents someone from typing their own consent phrase and then submitting a recording of a different person (e.g., a public figure, coworker, or family member) as the "sample."
13. **Bind the consent phrase and the voice sample to one continuous capture.** The consent-phrase recording and the additional voice sample must be captured back-to-back in a single uninterrupted browser recording session (no pausing to swap source audio), and both clips must be tagged with the same capture session ID. The backend must reject an audio-sample submission whose capture session ID doesn't match an existing, completed consent-phrase recording for that participant.
14. **Filter what the host is allowed to generate.** Before any generation request reaches the voice provider, run the input text through a content check that blocks/flags categories such as: financial authorization or payment instructions, one-time codes/PINs/passwords, requests to impersonate a named real-world public figure, urgent "send money now" / "wire transfer" phrasing, and impersonation of a person not in the session's participant list. Blocked attempts must be logged (not silently dropped) so the host dashboard can show "N generation attempts blocked by content policy" — this is itself a useful thing to show on camera.
15. **Embed the disclosure in the audio, not just the UI.** Every generated clip must open or close with a spoken or clearly audible tone-based disclosure (e.g., a short spoken line or a distinct audio watermark tone) stating the audio is synthetic and was generated for a cybersecurity education demo. A UI label alone isn't enough, since the clip can be clipped out of the app and shared without it.
16. **Auto-expire everything.** Consent records, audio samples, and generated clips must have an expiration policy (configurable TTL, default short — e.g., 24 hours) and must also be purged immediately when the host ends the session. No voice data should persist indefinitely by default.

## Build the MVP using

* Frontend: Next.js with TypeScript
* Backend: FastAPI with Python
* Database: SQLite
* Audio capture: browser MediaRecorder API only (no file input elements anywhere in the audio-capture flow)
* Voice generation: create a mock VoiceCloneProvider interface for now. The mock provider can return placeholder audio or use browser text-to-speech. Do not require a paid API key for the first version.
* Structure the code so a real consent-based voice provider could be added later, and so the content-filter step and audio-watermark step are provider-agnostic (they wrap any provider, not just the mock).

## Required user flow

1. Host creates a session, optionally setting a data-retention TTL (defaults to 24 hours).
2. Host manually adds participant names.
3. Each participant receives or opens a unique participant page.
4. Participant reviews a consent disclosure.
5. Participant must click an explicit consent button before recording.
6. Participant records, in one continuous browser-mic session: (a) the phrase "I consent to this voice sample being used for this cybersecurity education demo," then (b) a short additional voice sample — captured back-to-back, never as separate uploads.
7. Host dashboard shows all participants, consent status, audio sample status, and a running count of blocked generation attempts.
8. Host types a sentence.
9. Host selects a consenting participant.
10. The sentence passes through the content filter; if blocked, the host sees why and the attempt is logged.
11. App generates or simulates synthetic voice playback, with the spoken/tone disclosure embedded in the output audio.
12. App blocks generation for participants who have not consented, who revoked consent, or whose audio sample doesn't match a verified capture session.
13. App labels every output as synthetic audio generated for a cybersecurity education demo, both visually and in the audio itself.
14. Host can end the session at any time, which immediately purges all consent records, audio samples, and generated clips for that session.

## Required pages

* Voice generation page (shows content-filter result before generating)

## Required backend endpoints

* POST /sessions
* GET /sessions/{session_id}
* DELETE /sessions/{session_id} (host-triggered immediate purge of all session data)
* POST /sessions/{session_id}/participants
* POST /participants/{participant_id}/consent
* POST /participants/{participant_id}/revoke-consent
* POST /participants/{participant_id}/audio-sample (accepts only a payload tagged with a capture_session_id matching a completed consent-phrase recording; no generic file upload)
* DELETE /participants/{participant_id}/audio-sample
* POST /generate-voice (runs the content filter before calling the provider; logs blocked attempts)
* GET /health
